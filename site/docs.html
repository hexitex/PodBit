Installation & Setup

Installation & Setup

Podbit runs as a Node.js server with an embedded SQLite database - no external database, Docker, or cloud services required. The server runs on Windows, macOS, and Linux — either locally on your machine or on a remote server accessible over the network.

Prerequisites

Node.js 18+ (required)

LTS version recommended. Check with node --version.

Git (required)

To clone the repository. On Windows, Git Bash includes OpenSSL.

OpenSSL (optional)

For TLS certificate generation. Only needed for remote access with HTTPS.

LLM API Key (optional)

An API key for at least one LLM provider (OpenAI, Anthropic, local, etc). Configure in the Models page after setup.

Local Installation (Quickstart)

The simplest setup - everything runs on your machine with zero security friction.

# 1. Clone and install
git clone https://github.com/anthropics/podbit.git
cd podbit
npm install

# 2. (Optional) Run the setup wizard to customize ports, data dir, API keys
npm run setup:local

# 3. Start all services
npm run orchestrate

# Orchestrator (:4711) spawns and manages:
#   API Server  → http://localhost:4710 (REST API, GUI, synthesis engine)
#   Proxy       → http://localhost:11435 (OpenAI-compatible knowledge proxy)
#   GUI Dev     → http://localhost:4712 (Vite dev server)
#
# Minimal alternative (API server only, no proxy or health monitoring):
#   npm start

Open http://localhost:4710 in your browser. No password, no login - the GUI authenticates automatically via a localhost key handshake.

First Steps After Install
  1. Go to the Models page and register at least one LLM (add API key and assign subsystems)
  2. Seed your first knowledge via the Chat page or MCP tools
  3. Start the synthesis engine from the Dashboard

Remote Server Installation

For running Podbit on a remote machine, VM, or cloud server accessible over the network. This enables the full security stack: JWT auth, TLS, CORS lockdown, rate limiting. See Security & Remote Access for full details on the auth system.

Option A: Interactive Setup Script (recommended)
git clone https://github.com/anthropics/podbit.git
cd podbit
npm install

# Interactive wizard — configures .env and generates TLS cert
npm run setup:remote

# Or use the unified setup (prompts for local vs remote)
# npm run setup

# Start all services (recommended for remote — includes health monitoring)
npm run orchestrate

The script prompts for host binding, ports, TLS certificate generation, and CORS origins. It writes a .env file and optionally generates a self-signed certificate.

Option B: Manual .env Configuration
git clone https://github.com/anthropics/podbit.git
cd podbit
npm install

# Create .env
cat > .env << 'EOF'
# Bind to all interfaces (triggers remote mode)
HOST=0.0.0.0

# Ports (defaults shown — change if needed)
API_PORT=4710
PROXY_PORT=11435

# TLS — strongly recommended for remote access
PODBIT_TLS_CERT=data/tls/podbit.cert
PODBIT_TLS_KEY=data/tls/podbit.key

# CORS — restrict which origins can access the API
# PODBIT_CORS_ORIGINS=https://my-app.example.com

# Database encryption (optional) — requires: npm install better-sqlite3-multiple-ciphers
# PODBIT_DB_KEY=your-secret-encryption-key
EOF

# Generate self-signed TLS certificate
npm run generate-cert -- your-hostname.local

# Start all services with health monitoring and auto-restart
npm run orchestrate
After Starting in Remote Mode
  1. The orchestrator (:4711) spawns the API server (:4710), knowledge proxy (:11435), and GUI dev server (:4712)
  2. Open https://your-hostname:4710 in your browser
  3. Accept the self-signed certificate warning (or import it into your trust store)
  4. Set your admin password on the first-visit setup screen
  5. Log in with your password to access the GUI
  6. Configure LLM models on the Models page
  7. For proxy API clients, use your security key (found in Settings) as the API key

MCP (IDE) Integration

Podbit exposes an MCP server for integration with LLM IDE agents (Claude Code, Cursor, Continue, etc). Two modes are available depending on your deployment:

Local Mode (same machine)

Uses mcp-stdio.ts — direct database access, no network needed.

{
  "mcpServers": {
    "podbit": {
      "command": "npx",
      "args": ["tsx", "mcp-stdio.ts"],
      "cwd": "/path/to/podbit"
    }
  }
}
Remote Mode (server on another machine)

Uses mcp-stdio-remote.ts — forwards all tool calls to the remote API server over HTTP. Requires a Podbit clone on the client for the MCP SDK and script. Works on Linux, macOS, and Windows clients.

{
  "mcpServers": {
    "podbit": {
      "command": "npx",
      "args": ["tsx", "mcp-stdio-remote.ts"],
      "cwd": "/path/to/podbit",
      "env": {
        "PODBIT_API_URL": "https://your-server:4710",
        "PODBIT_API_KEY": "your-security-key"
      }
    }
  }
}

For self-signed TLS certs, add "NODE_TLS_REJECT_UNAUTHORIZED": "0" to the env block, or import the cert into the client OS trust store.

Client setup: Clone the repo on the client machine, run npm install, then use npm run setup:client on the server to generate pre-filled config snippets with your server URL and key.

Knowledge Proxy Client Setup

The knowledge proxy enriches any OpenAI-compatible request with knowledge graph context. Point any client at the proxy URL and every request is automatically enriched. Run npm run setup:client to generate ready-to-use config snippets for all platforms, pre-filled with your server settings.

Common Issues

Port already in use (EADDRINUSE)

Another process is using the port. Change it in .env (e.g., API_PORT=4720) or stop the conflicting process.

Cannot connect to GUI

Check that the server started successfully (look for the startup banner in terminal). If remote, ensure your firewall allows the API port.

Self-signed certificate warning

Expected with self-signed certs. Either accept the warning in your browser or import data/tls/podbit.cert into your OS/browser trust store.

openssl not found

Install OpenSSL: Windows (included with Git for Windows), macOS (brew install openssl), Linux (apt install openssl). Or provide your own cert files.

Models page shows no models

You need to register at least one LLM. Go to Models, click Add Model, and configure your provider's API key and endpoint.

What is Podbit?

What is Podbit?

Feed Podbit your research — papers, notes, code, raw ideas — and it discovers connections you didn't know existed. It pairs your knowledge, synthesizes new insights autonomously, verifies claims via external lab servers, and surfaces contradictions where breakthroughs hide. Everything lives in a persistent knowledge graph that grows smarter with every cycle.

Your First Steps

Your First Steps

This guide gets you from zero to your first synthesized insight. You need at least one LLM endpoint. By the end, you will have models registered, knowledge seeded, and the synthesis engine producing connections autonomously.

Step 1: Set Up Models

Podbit has 25+ subsystems, each assignable to a different model. The minimum to start is two assignments: embedding (local recommended) and synthesis (cloud recommended). Everything talks the OpenAI-compatible API. Unassigned subsystems simply don't run — the system gracefully skips cycles without a model.

Local Models — small, fast, cheap

Best for high-volume subsystems where throughput matters more than reasoning depth. Embeddings, autorating, keyword extraction, and domain classification run frequently and benefit from zero API cost.

  • LM Studio — GUI-based, easy setup, runs GGUF models
  • Ollama — CLI-based, auto-pulls models, lightweight
  • vLLM / text-generation-inference — high-throughput serving
  • Any server with an OpenAI-compatible /v1/chat/completions endpoint

Cloud APIs — medium & frontier quality

Synthesis, voicing, and research produce substantially better output with stronger models — cloud also gives you speed. A built-in budgeting system tracks spend and caps costs so you stay in control. Mix freely: cloud for quality-sensitive subsystems, local for volume.

  • OpenAI — GPT-4o, o1, o3-mini, etc.
  • Anthropic — Claude (via OpenAI-compatible wrapper)
  • Any OpenAI-compatible — Z.AI, Groq, Together, etc.
  • Per-model API keys supported in the model registry

Embeddings: Run Locally

Embeddings are the foundation — every node gets embedded for similarity search, synthesis pairing, dedup, and relevance scoring. A local embedding model performs well even on CPU and avoids per-call API costs that add up fast with thousands of nodes. Popular choices: nomic-embed-text, bge-m3, mxbai-embed-large.

Step 2: Start Podbit

The full stack includes the REST API, GUI, Knowledge Proxy, and synthesis engine. It can start in two ways:

Direct start (recommended)

npm run start

Starts the server. Open the GUI in your browser and configure models, seed knowledge, and start synthesis from the Dashboard.

Via MCP (auto-starts)

If you configure Podbit as an MCP server in your IDE, the stack auto-starts when the agent first connects. No manual startup needed — the MCP entry point boots everything.

Ports are configurable in .env. Requires Node.js 18+.

Step 3: Register Models

Open the Models page in the GUI. Register your LLM endpoints, then assign them to subsystems. For full details, see Configuration.

  1. Register models — Add your local or cloud endpoints. Each model needs a name, base URL, model ID, and provider type. Test the connection to verify.
  2. Assign embedding first — Assign your embedding model to the embedding subsystem. Without this, nodes can't be embedded and synthesis can't find pairs.
  3. Assign synthesis — Assign an LLM to the synthesis subsystem. This is the model that produces insights from node pairs.
  4. Optional extras — Assign models to voice, research, chat, and others as needed. Small local models for high-volume subsystems, frontier cloud models for quality-sensitive tasks.

Step 4: Add Knowledge & Start Synthesis

With models assigned, feed the graph and let Podbit think. For a deeper guide on seeding strategies, see Adding Knowledge.

  1. Add initial seeds — Use the GUI Chat, the Graph page, or Knowledge Base folder ingestion to add your starting material. 10-20 seeds in a domain is enough to begin.
  2. Start the synthesis engine — On the Dashboard, switch to API mode and click Start. The engine begins discovering connections between your seeds autonomously.
  3. Enable autonomous cycles — Toggle on voicing, research, questions, tensions, validation, and autorating cycles as desired. Each runs independently on its own interval.
  4. Watch the activity feed — The Dashboard shows real-time events as cycles run. You'll see pairings, rejections, new nodes, and quality gate decisions.
  5. Review and curate — Browse the Graph page to see what the engine produced. Rate nodes, promote breakthroughs, remove junk, and add more seeds to guide the research. See Growing the Graph for synthesis details.

Key Concepts

Key Concepts

Podbit organizes everything as nodes in a knowledge graph. Seven autonomous cycles run in parallel — some create new knowledge, some evaluate it, some destroy it. Understanding these building blocks helps you get the most out of the system.

Nodes

Everything in Podbit is a node in the knowledge graph. Each node carries content (the actual knowledge), a type, a domain label, a weight (importance), salience (attention priority), and an embedding vector for semantic similarity search.

seed

Raw input from you, the research cycle, or KB ingestion. The starting material for everything.

synthesis

Auto-generated connections between ideas, created by the synthesis engine via precision pairing.

voiced

Perspective-driven insights from the voicing cycle. Five persona modes: sincere, cynic, pragmatist, child, object-following.

breakthrough

Validated discoveries that passed scoring on synthesis quality, novelty, testability, and tension resolution.

question

Knowledge gaps identified by the question cycle. Guides future research and investigation.

raw

Verbatim file content from KB ingestion. Searchable reference material, excluded from synthesis.

elite_verification

Lab experiment results — verification data and supported/refuted status attached to the claims they tested.

For the full node type reference including weight mechanics and lifecycle details, see Node Types & Lifecycle.

The Seven Cycles

Each cycle has a distinct role. Three create new knowledge, four evaluate and stress-test what already exists, and one verifies claims empirically.

Creators — Make New Knowledge

Synthesis Engine

Finds hidden connections by pairing nodes in a tuned similarity band and running them through quality gates. Produces synthesis nodes.

Voicing Cycle

Explores ideas through five persona modes (sincere, cynic, pragmatist, child, object-following) using loose, random pairing. Produces voiced nodes.

Research Cycle

Fills knowledge gaps in sparse domains by generating new seed nodes, filtered through an embedding relevance gate to stay on-topic.

For a detailed comparison of how these creators differ, see Growing Your Graph.

Evaluators — Judge Existing Knowledge

Validation

Scores high-weight nodes on synthesis quality, novelty, and testability, promoting the best to breakthrough status.

Autorating

Automatically adjusts node weights by quality, creating natural selection pressure where good ideas propagate and weak ones fade.

Tensions

Finds pairs of highly similar nodes that make opposing claims, surfacing contradictions where unknown knowledge hides.

Questions

Generates specific, answerable research questions from node pairs, making knowledge gaps explicit and actionable.

Verifier — Test Claims Empirically

Lab Verification

Extracts experiment specs from claims and submits them to external lab servers. Labs run experiments and return raw data. Podbit evaluates the returned data against spec criteria. Nodes are frozen during verification and tainted downstream if refuted. The only cycle that tests claims empirically — the system's reality check. See Verification & Quality for details.

Domains & Partitions

Knowledge is organized by domains — labels like "biology", "architecture", or "synthesis-design" that group related nodes. Domains are then grouped into partitions, which control isolation boundaries.

Isolation

Nodes in different partitions cannot synthesize together unless the partitions are explicitly bridged. This prevents unrelated topics from producing nonsensical cross-domain synthesis.

Bridging

When two partitions should cross-pollinate, you create a bridge. Bridged partitions allow synthesis to pair nodes from different domains — this is where cross-domain insights emerge.

System partitions

Some partitions (e.g., know-thyself) are marked as system-internal. They synthesize only within themselves and cannot be bridged to user partitions.

Dynamic

Domains and partitions are created as needed. When you seed knowledge to a new domain, a partition is auto-created. You manage them on the Graph page.

For full details on managing domains, partitions, and bridges, see Projects & Domains.

Weight & Salience

Two scores determine how nodes participate in the graph over time:

Weight = Importance

Starts at 1.0. Breakthroughs get boosted to 1.5. Parent nodes gain weight when their children are created. Autorating adjusts weight up or down based on quality scoring. Higher-weight nodes are sampled more often for synthesis.

Salience = Attention

Gets a +0.1 boost when a node is matched in a query or synthesis. Decays by x0.99 each cycle. Determines how frequently a node is selected for pairing — recently active and relevant nodes bubble up, while stale ones fade.

For full weight and salience mechanics, see Node Types & Lifecycle.

How They Work Together

The cycles form a self-reinforcing ecosystem. Each step feeds into the next:

  1. You seed initial knowledge (manually, via KB, or via Chat)
  2. Research fills gaps in sparse domains so synthesis has enough material
  3. Synthesis finds precise, validated connections between ideas
  4. Voicing explores the same ideas through creative, adversarial, and naive perspectives
  5. Questions identify what's still unknown, guiding future research and seeding
  6. Tensions surface contradictions — the most productive areas to investigate
  7. Autorating adjusts node weights so the best ideas propagate through future synthesis
  8. Validation promotes the highest-quality insights to breakthrough status
  9. Lab verification empirically tests claims via external lab servers, adding verification scores that feed back into quality

All cycles run in parallel on configurable intervals. For details on cycle configuration and the synthesis pipeline, see Growing Your Graph. For input paths and how to add knowledge, see Adding Knowledge.

Adding Knowledge

Adding Knowledge

Knowledge enters Podbit through six paths, each suited to a different source. You can manually seed ideas, bulk-import documents, let LLMs fill gaps, or feed in live data from external APIs.

Six Input Paths

Manual Seeding

The most direct path. Use the GUI Chat in seed mode, the Graph page, or type ideas directly. You control exactly what enters the graph. Seeds carry your domain expertise, hypotheses, observations, and curated facts. IDE agents can also seed via MCP.

Creates: seed nodes. Contributors: human:*, claude

Knowledge Base (File Ingestion)

Bulk import from local folders — papers, documents, code, spreadsheets, images. Six reader plugins process files and create nodes. Curated mode uses LLMs to generate descriptions as seed nodes. Raw mode ingests verbatim as raw nodes (searchable, but excluded from synthesis).

Creates: seed or raw nodes. Contributors: kb:*

Research Cycle (LLM Knowledge)

The research cycle asks an LLM to generate factual seeds for under-populated domains, drawing on the model's training knowledge. This fills gaps with the model's broad knowledge about a topic, filtered through relevance gates so only on-topic content makes it in.

Creates: seed nodes. Contributors: research-cycle

API Enrichment (External Data)

Registered external APIs (weather services, scientific databases, financial data, etc.) are called during lab verification. The API response is parsed by an LLM into discrete facts, and each fact becomes a new node. This is how real-world data enters the graph — from live sources, not from the LLM's training.

Creates: seed nodes. Contributors: api-enrichment:*

Synthesis & Voicing (Derived Knowledge)

Not technically an "input" — these cycles recombine existing knowledge into new insights. But their output feeds back into the graph and becomes input for future cycles. A synthesis node can be paired with a seed to produce a second-generation synthesis, and so on. This is how the graph deepens over time.

Creates: synthesis and voiced nodes. See Growing the Graph.

Chat & Proxy (Conversational)

The GUI Chat and Knowledge Proxy can seed knowledge from conversations. When a chat response contains useful information, it can be proposed to the graph. The proxy enriches LLM calls with graph context, and the resulting conversation can feed discoveries back.

Creates: seed nodes. See Chat & Questions.

Knowledge Base Deep Dive

The Knowledge Base (KB) ingests local folders into the knowledge graph via a plugin-based reader pipeline. It handles everything from plain text to images, with smart chunking, change detection, and optional real-time file watching.

How It Works

  1. Add a folder — Register a local folder path with a target domain. Configure include/exclude glob patterns, recursive scanning, and whether to use raw or curated mode.
  2. Scan — The scanner recursively walks the folder, matching files against glob patterns. Each file is hashed (SHA-256) for change detection.
  3. Read — Matched files are dispatched to the appropriate reader plugin based on file extension. Each reader extracts text content and splits it into smart chunks.
  4. Ingest — The pipeline embeds each chunk, creates nodes in the graph, and links child chunks to their parent file node via edges. Domain is derived from the folder path.
  5. Watch — If file watching is enabled, chokidar monitors the folder for changes. Modified files are automatically re-scanned. Unchanged files (same content hash) are skipped.

Key Features

Change Detection

Every file is hashed with SHA-256 on scan. Re-scans skip unchanged files entirely — only modified or new files are re-processed. This makes re-scans fast even for large folders.

Domain Mapping

Each folder maps to a domain in the graph. Subfolders automatically inherit the parent domain unless overridden. This keeps ingested knowledge organized by source location.

File Watching

Optional chokidar-based real-time monitoring. When a watched file changes on disk, it is automatically re-scanned and re-ingested without manual intervention.

Pipeline Control

Concurrency limits prevent overwhelming the system during large ingestions. Failed files can be retried individually or in bulk. The pipeline queue processes files in order with configurable parallelism.

Raw Mode

When a folder is registered with rawMode: true, files are ingested verbatim as raw nodes. No LLM curation is applied — the content is stored exactly as-is.

Raw nodes are excluded from all autonomous cycles — synthesis, voicing, tensions, validation, questions, and research will never touch them. They exist solely as searchable reference material for RAG-style retrieval via podbit.query and podbit.compress.

Use for: Large reference corpora, source code archives, documentation sets, or any content you want searchable but not synthesized. Embeddings are prefixed with the filename for improved retrieval relevance.

Dual-folder pattern: Register the same source folder twice — once in raw mode (domain: my:raw) and once in curated mode (domain: my:curated). This gives you both verbatim source content for precise retrieval and curated knowledge for graph synthesis.

Growing Your Graph

Growing the Graph

Once you've seeded knowledge, Podbit's autonomous cycles take over. They run in parallel, each discovering a different facet of your research: connections, contradictions, gaps, and quality signals.

Starting the Engine

The synthesis engine and its cycles are controlled from the Dashboard. Once started, they run continuously on configurable intervals until you stop them.

Two Operating Modes

API mode — the engine calls LLMs directly to generate synthesis, voicing, and research output. Fully autonomous. MCP mode — the engine queues discoveries as pending requests for your IDE agent (Claude, Cursor, etc.) to process. Use MCP mode when you want human-in-the-loop control or when your IDE agent has better model access.

Pipeline Mode

Heuristic — all mechanical quality gates active (100+ tunable parameters). Maximum control, predictable behavior. Consultant — a single comprehensive LLM call replaces five meaning-judgment gates (claim provenance, hallucination detection, counterfactual independence, derivative check, fitness grading). Simpler, but depends on model quality.

Cycle Controls

Each cycle can be independently toggled on or off. Start with synthesis and voicing enabled, then add research, questions, tensions, validation, and lab verification as your graph grows. Cycles that need sparse domains (research) or high-quality nodes (validation) benefit from waiting until there's enough material.

Activity Feed

The Dashboard shows a real-time activity stream via Server-Sent Events. Every synthesis attempt, gate rejection, voicing result, and cycle event appears as it happens. Filter by category (synthesis, voicing, kb, proxy, etc.) to focus on what matters.

MCP Mode — Human-in-the-Loop Synthesis

In MCP mode, the engine finds resonating node pairs but does not call an LLM itself. Instead, it queues discoveries for your IDE agent (Claude Code, Cursor, Windsurf, etc.) to process. This lets you choose which model voices each insight and review results before they enter the graph.

  1. Engine runs — samples pairs, computes similarity, queues matches above threshold
  2. Check for discoveries: podbit.pending
  3. For each pair, call podbit.voice(nodeId) to get voicing context
  4. Read both nodes, synthesize an insight connecting them
  5. Save: podbit.propose(content, nodeType: "voiced", parentIds: [...])
  6. Mark complete: podbit.complete(requestId, result)

AI Research Supervisor Pattern

Any MCP-compatible agent (Claude Code, Cursor, Windsurf, VS Code agents) can act as a research supervisor — orchestrating synthesis, validating outputs, tuning parameters, and curating the knowledge graph. Frontier models are recommended for supervisory roles.

Supervised Synthesis

Run the engine in MCP mode and let the AI agent voice discoveries. Review proposed nodes before they enter the graph. Reject or refine low-quality outputs with the agent's help. This produces higher-quality synthesis than fully autonomous API mode.

Autonomous Tuning

Ask the agent to read metrics (podbit.config(action: "metrics")), diagnose issues, tune parameters, and monitor results. The agent can run observe-hypothesize-tune-verify loops across the full config surface.

Graph Curation

Have the agent find tensions (podbit.tensions), generate research questions, validate breakthrough candidates, dedup the graph, and provide feedback on node quality — all through MCP tools.

Cross-Domain Discovery

Use podbit.patterns to find abstract connections across domains. The agent can identify structural patterns that bridge unrelated knowledge areas, tag nodes, and synthesize cross-domain insights.

The Three Knowledge Creators

Three cycles generate new nodes. Each answers a fundamentally different question about your knowledge.

Synthesis Engine — "What connections exist between these ideas?"

The main engine. It's the most rigorous cycle and produces the highest-quality output because it's the most selective about what it combines and the most aggressive about filtering bad results.

How it pairs nodes

Samples a node by salience (a score combining weight, recency, and randomness), then uses embedding similarity to find the best partner within a tuned resonance band — not too similar (would produce tautologies), not too different (no meaningful connection). This is a precision match.

Quality gates (6+)

After the LLM voices the connection, output passes through: claim provenance (is it grounded in the parents?), hallucination detection, counterfactual independence (do both parents genuinely contribute?), redundancy ceiling, dedup, specificity scoring, and fitness grading. Most synthesis attempts are rejected.

Why it exists

Synthesis is the core value proposition of Podbit — finding non-obvious connections between ideas that a human wouldn't think to put together. The strict quality pipeline ensures these connections are genuinely meaningful, not just superficially related. It always uses the object-following persona (neutral, analytical) because the goal is to discover what's actually there, not impose a perspective.

Birth Pipeline — Mechanical Gates + Minitruth

Every synthesis output passes through fast mechanical checks (resonance, structural, specificity, dedup, junk) followed by minitruth, a single LLM reviewer that accepts, reworks, or rejects. The cull pipeline runs separately to evaluate existing nodes via a comprehensive LLM consultant.

Voicing Cycle — "What would a different perspective see in these ideas?"

The creative complement to synthesis. Where synthesis asks "what's objectively connected?", voicing asks "what would a cynic/pragmatist/child notice that the analyst missed?"

How it pairs nodes

Deliberately loose. Picks a high-weight node, then grabs a partner from its parents or random high-weight nodes. No embedding similarity search, no resonance band. This randomness is intentional — unexpected pairings produce unexpected insights.

Persona modes

Each run picks a random persona: sincere (empathetic, values-driven), cynic (adversarial, finds weaknesses), pragmatist (practical, implementation-focused), child (naive, asks obvious questions), object-following (neutral, analytical). Each sees different things in the same pair of ideas.

Why it exists separately from synthesis

Synthesis optimizes for correctness — rigorously validated, analytically neutral connections. But research breakthroughs often come from asking a dumb question, playing devil's advocate, or looking at something from a completely different angle. The voicing cycle provides this diversity of thought. It has fewer quality gates (no claim provenance, no counterfactual check, no fitness grading) because creative perspectives shouldn't be held to the same standard as analytical derivations. The graph benefits from both: precise connections and wild ideas.

Research Cycle — "What's missing from this domain?"

Unlike synthesis and voicing which recombine existing knowledge, research adds entirely new information. It targets under-populated domains and asks an LLM to generate factual seeds that fill gaps.

Domain targeting

Finds the domain with the fewest nodes (most in need of research). Validates it against the project manifest to prevent off-topic generation. Skips exhausted domains where recent cycles produced nothing new — domains naturally saturate.

Relevance filtering

Each generated seed is embedded and compared to the domain centroid (average embedding of existing nodes). Off-topic seeds are rejected. This prevents the LLM from drifting into tangentially related territory.

Why it exists

Synthesis and voicing can only recombine what's already in the graph. If a domain has 5 nodes, there are limited pairings. Research bootstraps sparse domains so synthesis has enough material to work with. It also brings in knowledge the human may not have thought to seed — the LLM fills blind spots. Research creates seed nodes (raw input), not voiced nodes (synthesized output), so they enter the graph at the ground level and participate in future synthesis.

The Four Evaluators

Four cycles don't create new knowledge — they evaluate, score, and stress-test what's already there. Without them, the graph would fill with untested claims and unresolved contradictions.

Validation Cycle

Evaluates high-weight nodes for promotion to breakthrough status. Scores them on synthesis quality, novelty, testability, and tension resolution. Breakthroughs are the graph's most validated insights — they've survived synthesis, quality gates, and explicit validation.

Why: Not all synthesized nodes are equal. Validation separates genuinely novel insights from competent-but-ordinary connections.

Autorating Cycle

Automatically scores nodes on quality, adjusting their weight. Higher-weight nodes get sampled more often by synthesis and voicing. Lower-weight nodes gradually fade. This creates a natural selection pressure where good ideas propagate and weak ones don't.

Why: Manual rating doesn't scale. Autorating lets the graph self-organize by quality without requiring human attention on every node.

Tensions Cycle

Finds pairs of nodes that are highly similar but make opposing claims. These contradictions are where unknown knowledge hides — if two well-supported ideas disagree, resolving the tension often produces a genuine insight.

Why: Contradictions are invisible in a flat list. The tensions cycle surfaces them automatically, turning the graph into a research tool that points you at the most productive questions.

Question Cycle

Examines node pairs and generates research questions — specific, answerable questions that neither node addresses. These question nodes guide the research cycle and help you identify what to investigate next.

Why: Knowing what you don't know is half the battle. The question cycle makes knowledge gaps explicit and actionable, rather than leaving them as vague feelings of incompleteness.

Lab Verification

Extracts experiment specifications from claims and submits them to external lab servers for empirical testing. Labs design experiments, run code, and return raw data. Podbit evaluates the data against the experiment spec's evaluation criteria — no LLM, just data comparison — and updates the node's verification status.

Lab verification is the system's reality check — the only cycle that tests claims against something other than other claims. Nodes are frozen during verification and tainted downstream if refuted. Not every claim is reducible to an experiment, but for those that are, empirical data is the strongest signal.

For a deep dive into verification methodology, code generation, and quality scoring, see Verification & Quality.

All Cycle Types at a Glance

Eight cycle types run in parallel. Each serves a distinct role in the knowledge lifecycle and can be independently enabled, disabled, and tuned.

Synthesis

Pairs nodes by embedding similarity, voices connections, creates child synthesis nodes. The primary knowledge generation cycle.

Autonomous Voicing

Persona-driven synthesis across 5 modes (object-following, sincere, cynic, pragmatist, child). Loosely paired, creative complement to analytical synthesis.

Validation

3-gate breakthrough pipeline: composite scoring → novelty gate (frontier model) → lab verification check. All gates fail-open.

Tensions

Finds contradicting node pairs with high similarity but opposing claims. Seeds research questions.

Questions

Generates research questions from tension pairs and knowledge gaps. Creates question-type nodes.

Autonomously generates foundational domain knowledge. Requires a project manifest. Embedding relevance gates reject off-topic seeds.

Autorating

Autonomous quality scoring on intake. Adjusts node weights to feed natural selection through the synthesis cycle.

How They Work Together

The cycles form a self-reinforcing ecosystem. Each serves a role that the others can't fill:

  1. You seed initial knowledge (manually, via KB ingestion, or via the GUI Chat)
  2. Research fills gaps in sparse domains so synthesis has enough material
  3. Synthesis finds precise, validated connections between ideas
  4. Voicing explores the same ideas through creative, adversarial, and naive perspectives
  5. Questions identify what's still unknown, guiding future research and seeding
  6. Tensions surface contradictions — the most productive areas to investigate
  7. Autorating adjusts node weights so the best ideas propagate through future synthesis
  8. Validation promotes the highest-quality insights to breakthrough status
  9. Lab verification empirically tests claims via external lab servers, adding verification scores that feed back into quality

All cycles run in parallel on configurable intervals. Each has its own LLM subsystem assignment and can be independently enabled, disabled, or tuned in Configuration.

Whether you use the GUI Chat or the Knowledge Proxy, the same enrichment pipeline runs behind the scenes:

Sessions map stateless OpenAI-compatible requests to stateful context engine sessions that accumulate topics and improve over turns. Session ID resolution order:

How Knowledge Selection Works

Every chat turn and proxy request runs through the context engine. A prepare() call selects and injects relevant graph knowledge before the LLM sees your message. An update() call runs after the response to boost nodes the LLM actually used and compute quality metrics.

Prepare Flow

  1. Detect intent — classify query as retrieval, action, diagnosis, or exploration
  2. Extract topics from user message + session history
  3. Cluster topics — group by embedding similarity into concept clusters
  4. Match domains from extracted topics
  5. Score nodes using 5 signals: embedding, topic, weight, cluster centroids, recency
  6. Adjust weights by intent (retrieval boosts embedding, diagnosis boosts recency)
  7. Apply model profile — adapt budget and node limits for model size
  8. Build system prompt (compressed format for small models)
  9. Use dynamic budget — more knowledge early, more history later
  10. Return context package with intent and metrics

Update Flow

  1. Track LLM response in session history
  2. Extract topics from response
  3. Feedback loop — compare response embedding to delivered nodes
  4. Boost weight of nodes the LLM actually used (similarity ≥ 0.65)
  5. Quality metrics — compute utilization, grounding, coverage, efficiency
  6. Store per-turn composite quality score

The feedback loop creates natural selection: useful knowledge gets boosted, unused knowledge slowly decays.

Intent Detection

Zero-cost keyword classifier — pure regex, no LLM call. Detects query intent and adjusts knowledge selection strategy.

Cross-Session Learning

Sessions are in-memory. When a session expires, its most valuable insights are persisted to the database so future sessions warm-start from your past conversation patterns.

What Gets Persisted

  • Top 30 topics with weights
  • Concept cluster terms
  • Node usage (which nodes were helpful)
  • Domain associations

How It Bootstraps

  • First turn loads matching insights
  • Topics merged at 30% weight (dampened)
  • Frequently-used nodes prioritized
  • EMA weight merging across sessions

Limitations

  • Token usage — The usage field in responses comes from the upstream LLM. If the provider doesn't return it, values will be 0.
  • Context window cost — Knowledge enrichment adds to the system message. The context engine uses dynamic budgets to limit this.
  • Provider compatibility — Not all providers support all parameters (e.g., local models may not support logprobs).

Breakthroughs

Breakthroughs are your graph's most validated discoveries — nodes promoted after scoring well on synthesis quality, novelty, testability, and tension resolution. They represent the highest-confidence insights the system has produced.

Breakthrough Lifecycle

1

Node created through synthesis, seeding, or voicing

2

Validation cycle or human identifies it as significant

3

May be marked as "possible" (pre-breakthrough candidate)

4

Validation scores computed across 4 dimensions (each 0-10)

5

If composite score meets threshold, promoted to breakthrough (weight set to 1.5)

6

Parents gain +0.15 weight, grandparents gain +0.05 — rewarding the ideas that contributed to a breakthrough

Validation Scores

Each breakthrough candidate is scored on four dimensions. Scores range from 0 to 10, with color coding: green (7+), yellow (5-6), red (<5).

Synthesis (0-10)

How coherently are the source ideas combined? Does the output represent a genuine integration, or just a surface-level concatenation?

Novelty (0-10)

Does this produce new understanding beyond what the source nodes already say? A high novelty score means the synthesis revealed something neither parent contained alone.

Testability (0-10)

Does the node make specific, verifiable claims? Vague philosophical observations score low. Concrete predictions or falsifiable statements score high.

Tension Resolution (0-10)

Does the node resolve contradictions between its source nodes? Tension-resolving insights are especially valuable because they advance understanding in areas of genuine disagreement.

Feedback & Rating

Rating nodes is how you shape the graph's evolution. Your feedback directly influences which nodes get sampled for future synthesis and which fade into low priority.

Useful

Thumbs up. Increases the node's weight, making it more likely to be sampled by synthesis and voicing. Signals that the insight is worth building on.

Not Useful

Thumbs down. Decreases the node's weight, reducing its sampling priority. The node remains in the graph but participates less in future cycles.

Harmful

Flags the node as actively misleading or incorrect. Significantly reduces weight and signals to the quality system that this content should not propagate.

Feedback can come from three sources: human (your manual ratings), agent (MCP tools via podbit.feedback), or autorating (the autorating cycle). All three affect node weight and future sampling.

Curating Your Graph

Active curation keeps your graph healthy. As the synthesis engine produces output, some nodes will be low-quality, redundant, or off-topic. Regular curation ensures the graph stays focused and useful.

Remove Junk

Archive removes a node from active participation but keeps it in the database for reference. Junk permanently filters the node and prevents similar content from being created in the future — the junk filter uses embedding similarity to block new nodes that resemble junked ones. Use junk only when you're certain the content is wrong or irrelevant, not just low-quality. Available in the GUI or via podbit.remove.

Dedup

Find and archive duplicate or near-duplicate nodes. The dedup algorithm clusters nodes by embedding similarity and word overlap. Always run a dry-run first to review the clusters before archiving — nodes that look similar in truncated previews may describe different concepts. Available via podbit.dedup.

Promote

Manually promote exceptional nodes to breakthrough status. This bypasses the validation cycle's automatic scoring — useful when you recognize an insight that the scoring system might not fully appreciate. Promoted nodes get weight 1.5 and their parents receive weight bonuses. Available via podbit.promote.

Edit

Modify a node's content text directly. Useful for correcting minor errors, improving clarity, or removing irrelevant details from an otherwise good synthesis. Editing re-embeds the node automatically so similarity searches stay accurate.

Rate

Thumbs up or thumbs down on node cards to adjust weight signals. Quick ratings are the lowest-effort way to shape the graph — even a few ratings per session meaningfully influence which ideas propagate through future synthesis cycles.

Curation Tips

  • Review after synthesis runs. Check the Activity Log to see what was produced, then browse the Graph page filtered to recent nodes.
  • Junk sparingly. The junk filter is powerful — it blocks similar future content via embedding similarity. Archive when unsure; junk only when certain.
  • Use lineage to understand context. Before removing a node, check its children. A mediocre node might be the parent of a strong synthesis.
  • Rate liberally. Even brief thumbs-up/down sessions help the autorating cycle calibrate and improve the quality of future synthesis.
  • Check verification status. Nodes that have been lab-verified carry stronger quality signals. Prioritize reviewing unverified breakthroughs.

Dual-Database Architecture

Podbit runs two concurrent SQLite databases. The system database (system.db) is permanent and shared across all projects — it stores models, prompts, subsystem assignments, and configuration history. The project database is per-project and holds the knowledge graph itself: nodes, edges, partitions, KB files, and all project-specific data.

The system database stays open — your model assignments, prompts, and configuration are untouched. Only the project database is closed and the new one opened. All caches are cleared, the synthesis engine reinitializes, and KB watchers restart for the new project's folders. SQLite uses WAL (Write-Ahead Logging) mode for concurrent read/write safety.

Number Variables

The number variable system prevents synthesis from universalizing domain-specific numbers across unrelated domains. For example, "1-5% activation density" from biology should not become a universal constant when synthesized with physics knowledge.

How It Works

1. Extraction: When a node is created, all numeric values are extracted and stored in a registry.

2. Placeholder replacement: Numbers in node content are replaced with [[[PREFIX+nnn]]] placeholders. The prefix is derived from a per-installation UUID, making variable IDs globally unique.

3. Resolution: Before any LLM receives node content, placeholders are resolved back to actual values. Voicing uses a different path — it keeps placeholders but injects a variable legend so the LLM sees each value's domain provenance.

source_node_idThe node this number was extracted from

Journal & Rollback

Every structural change to the knowledge graph is automatically recorded by SQLite triggers into an undo journal. Unlike traditional database rollback which is all-or-nothing, Podbit's journaling is purpose-built for knowledge graphs — it lets you roll back to any point in time while selectively preserving valuable nodes. Pin a breakthrough and its entire parent ancestry is automatically exported, the rollback executes, and the pinned lineage is reimported with original timestamps intact. This means you can undo a bad KB ingestion or runaway synthesis cycle without losing genuine insights that emerged from it.

What Gets Journaled

Structural changes only — node creation, deletion, content edits, type changes, domain moves, archive/junk status, verification results, and lab status. Scoring metadata (weight, salience, barren cycles, lifecycle state) is not journaled — it is recalculated by the synthesis engine and would generate thousands of noise entries per cycle.

Journaled Tables

nodes, edges, domain_partitions, partition_domains, partition_bridges, number_registry, node_number_refs

Not Journaled

activity_log, resonance_cycles, lab_executions, knowledge_cache, config_history, chat sessions, feedback

Timeline & Rollback Flow

The Journal section on the Data page shows journal entries grouped into time buckets with human-readable descriptions (e.g., “3 nodes created in Optimization Algorithms”, “2 edges added”).

  1. Hover any time bucket and click Restore to preview what would be undone
  2. The preview shows nodes that will be removed, reverted, or re-created — with a per-table breakdown
  3. All pinnable nodes (voiced, synthesis, possible, elite, breakthrough) created after the restore point are listed with checkboxes
  4. Select nodes you want to keep, then click Pin & Restore
  5. The system exports pinned nodes plus their full parent ancestry, executes the rollback, then reimports the pinned package with original IDs and timestamps

Seeds are not pinnable — they are automatically captured as ancestors of any pinned node. All background services (synthesis, voicing, research cycles) are stopped before rollback executes.

Pinning & Preservation

Pinned nodes retain their original created_at and updated_at — they are reimported with INSERT OR IGNORE using original GUIDs. On a timeline, a preserved node from days ago sitting alongside new nodes visually shows it survived a rollback.

Clipping

The Clip button trims journal entries from the oldest end. There is no automatic retention policy — if you load a project dormant for months, its journal is preserved until you manually clip it. Clipping is irreversible: you cannot roll back past a clipped point.

Verification & Quality

Verification and Quality

Podbit uses a two-pipeline architecture to balance creative freedom with quality control. The Birth pipeline is permissive — fast mechanical checks plus minitruth, a single LLM reviewer. The Cull pipeline runs periodically to evaluate existing nodes and prune weak ones. Beyond these, the Lab framework submits claims to external lab servers for empirical testing, and the elite pool distills the most validated knowledge through generational synthesis.

Two-Pipeline Architecture

Every synthesis output passes through the birth pipeline before entering the graph. Once in the graph, nodes are periodically re-evaluated by the cull pipeline. This separation lets creative cross-domain connections form freely, while still maintaining graph quality over time.

Birth Pipeline

Fast mechanical checks followed by minitruth, a single LLM reviewer. The mechanical gates are deterministic math/string operations — no LLM calls. Minitruth makes the final meaning judgment: accept, rework, or reject.

  • Mechanical: Resonance, Structural, Voicing, Specificity, Dedup, Junk
  • Minitruth: Single LLM call — accept / rework (re-voice with feedback) / reject

Design: Permissive creation — let creative connections through, judge meaning once.

Cull Pipeline

Periodically evaluates existing nodes using a single comprehensive LLM call per node. The consultant scores each node across five weighted dimensions and the composite score determines the outcome: boost, demote, or archive.

  • Coherence (30%) — does the node make logical sense?
  • Grounding (25%) — is it grounded in its parent nodes?
  • Novelty (20%) — does it say something new?
  • Derivation (15%) — are specific claims derived from reasoning, not just inherited?
  • Forced Analogy (10%) — is the connection genuine?

Design: Strict quality sweep — nodes must prove their value to remain in the graph.

Birth Pipeline Gates

The birth pipeline runs these gates in order. Mechanical gates are deterministic — no LLM involved. Minitruth is the only LLM call in the birth pipeline.

Mechanical Gates

Deterministic math and string operations. Every synthesis output must pass all of them before reaching minitruth.

Resonance Band

Embedding similarity between parent nodes must fall within a tuned range. Too similar produces tautologies; too different produces nonsense.

Structural Validation

Ensures the output is well-formed — proper length, not empty, meets format expectations. Catches malformed LLM responses before they enter the graph.

Voicing

The LLM synthesis call itself — generating the insight from parent nodes. Includes hallucination heuristics, derivative checks, and truncation detection.

Specificity Scoring

Measures concreteness — penalizes vague, generic language. A synthesis that says "these ideas are related" adds nothing. Specificity scoring demands substance.

Dedup

Embedding similarity plus word overlap detection. Catches duplicates via rephrasing. Optional LLM judge for borderline cases in the doubt zone.

Junk Filter

Compares against embeddings of previously junked nodes. Blocks content similar to known-bad output. Has 30-day decay to prevent permanent topic blocking.

Minitruth (LLM Reviewer)

After all mechanical gates pass, a single LLM call evaluates the synthesis. Armed with the project manifest for domain context, minitruth judges whether this content deserves to exist in the knowledge graph.

Accept

Content is valuable and well-formed. Proceeds to node creation.

Rework

Has potential but needs improvement. The feedback is sent back to voicing for a second attempt (max 1 rework). If rework still fails, the synthesis is rejected.

Reject

Content is not worth keeping — hallucinated, derivative, or irrelevant. Discarded permanently.

Breakthrough Validation Pipeline

Nodes that survive synthesis quality gates can be further elevated to breakthrough status — the graph's highest tier of validated knowledge. Promotion goes through a dedicated 3-gate pipeline. View breakthroughs on the .

Three Validation Gates

1. Composite Scoring

Four dimensions scored: synthesis quality, novelty, testability, and tension resolution. The composite score must exceed a threshold for promotion.

2. Novelty Gate

Is this insight genuinely new? Checks whether the knowledge already exists elsewhere in the graph at a higher quality level. Prevents promoting redundant breakthroughs.

3. Lab Verification Gate

A final check against lab experiment results. If a lab has tested this claim and it was refuted, it cannot become a breakthrough regardless of other scores.

Post-Promotion Effects

When a node is promoted to breakthrough: its weight rises to 1.5 (making it more likely to be sampled in future synthesis), its parent nodes get +0.15 weight (rewarding the ideas that led to the breakthrough), and grandparent nodes get +0.05. This creates a reinforcement signal that propagates backward through the graph — good ideas that produce breakthroughs are recognized automatically.

Lab Verification

Labs are Podbit's reality check. Claims are submitted to external lab servers for empirical testing — math computations, neural network training runs, parameter sweeps, simulations, or any experiment the lab can run. The three-stage pipeline isolates bias: only the spec extractor sees the claim narrative, the lab only sees experiment specifications (not the original claim), and the evaluator interprets raw data without narrative access. Claims that can't be reduced to testable experiments are honestly tagged as "not reducible" — no fake verification. Use MCP tools or the Verification page in the GUI.

Three-Stage Pipeline (Bias-Isolated)

1. Spec Extraction + Review (Podbit)

The ONE place that reads the claim narrative. An LLM extracts a structured experiment specification — what to compute, what to measure, what tolerance to use. Claims that can't be reduced to a testable spec are honestly tagged "not reducible." Prior rejection reasons are injected to prevent flip-flopping. The spec then passes a structural tautology check (rejects embedded code) and an optional adversarial falsifiability review (a second LLM checks for cherry-picked parameters). This is the auditable bias surface.

2. Lab Execution (Separate Server)

The lab receives a spec with experiment setup and measurements — never the claim. It generates code (using its own LLM), runs the experiment, and returns raw data plus any file artifacts (plots, logs, metrics). Labs are separate projects with their own config, models, and sandbox. Podbit talks to them over HTTP.

3. Data Evaluation (Podbit)

Evaluates lab results against the spec's evaluation criteria using the spec's evaluation criteria: numerical identity (within tolerance), convergence (series behaviour), threshold (above/below bound), trend (directional relationship), or boolean (pass/fail). Confidence is derived from data, not hallucinated by an LLM.

Node Freezing

When a node enters lab verification, it is frozen — excluded from synthesis pairing, decay, and lifecycle sweeps. This prevents the graph from building on unverified claims. The node is unfrozen when the experiment completes, regardless of outcome.

Taint Propagation

When a claim is refuted, its downstream children can be tainted — marked as potentially unreliable because their foundation was disproved. Tainted nodes are excluded from synthesis until the taint expires (configurable decay) or is cleared by re-verification. Enable via config.

Lab Queue

Nodes can be enqueued for lab verification manually (via MCP or GUI) or automatically by the verification cycle. The queue worker freezes each node, submits the experiment spec to the lab server, polls for completion, then evaluates the raw data and applies graph consequences.

Each queue entry carries a template ID — different lab servers handle different experiment types.

Evidence Commons

Lab results are stored as reusable evidence — not just verification logs. Computed values, metrics, and file artifacts (plots, logs, model weights) are stored in the evidence table and can be referenced by multiple nodes. A sparsity measurement from one experiment can be cited by any node discussing sparsity.

Decompose: Broad claims can be split into atomic facts for individual verification.

Lab Architecture

Labs are separate projects — they run their own servers, have their own config, models, and sandbox rules. Podbit talks to them over HTTP. The quick-verify lab handles mathematical, computational, and claims. Other labs (NN training, simulation) handle different experiment types. Each lab implements the same 3-endpoint contract: submit, status, result.

Separation of Concerns

Podbit's Job

  • Decide whether to test (weight thresholds, queue)
  • Extract experiment specs from claims (the bias surface)
  • API reconnaissance (pre/post-lab fact checking)
  • Evaluate lab data against spec criteria
  • Apply graph consequences (weight, taint, archive)
  • Store evidence in the empirical commons

Lab's Job

  • Generate code from experiment specs (own LLM, own language)
  • Run experiments in sandbox (own security policy)
  • Collect artifacts (plots, logs, metrics)
  • Return raw data — not verdicts
  • Handle retries internally
  • Serve artifacts via HTTP

Lab URL Configuration

Labs are reachable at HTTP URLs that the Podbit API needs to know. There are two kinds of labs and they store their URLs differently — built-in / co-located labs are config-driven so you can change ports without touching the database, while remote labs (added through the GUI) keep a literal URL in the registry because there's nowhere else to put it.

Built-in / Co-located Labs

math-lab, nn-lab, critique-lab — they ship in the podbit-labs/ sibling repo and run on the same machine. They declare a port key (e.g.mathLab) that binds them to a slot in config/port-defaults.json.

The registry stores the URL as a hint only — every read overlays the live URL fromPORTS[port_key]. Changing a port in .env propagates with no DB edits, no migrations.

Set port_key when registering the lab via the Labs page — the URL field becomes a hint and is ignored.

Remote / User-added Labs

Labs you add via the Labs page that point at an external service — a colleague's cluster, a cloud VM, a CI runner. There's no config slot for them, so the URL you type into the form is the source of truth and is stored verbatim in the registry.

Leave the Port Key field blank when registering. If the URL ever changes, edit it in the GUI — the registry row is the only place it lives.

Auth (bearer token, API key, custom header) is also stored in the registry alongside the URL — only remote labs typically need it.

API Reconnaissance

External APIs provide real-world data for fact checking and enrichment. API reconnaissance runs in two phases: pre-lab (before spec extraction — corrects facts so the spec extractor works with verified data) and post-lab (after evaluation — enriches verified nodes with additional context from live data sources).

Verify Mode

Check claims against external data sources. The API response is compared to the node's claims — does the real data support or contradict the synthesis?

Enrich Mode

Add context from external data. Doesn't check claims — instead augments the node with additional facts, statistics, or details from live data sources.

Both Mode

Verify and enrich in one pass. Checks existing claims against external data while also adding new contextual information from the API response.

Intake Defense

Beyond per-synthesis quality gates, Podbit has system-level defenses that protect the graph from structural problems.

Domain Concentration Throttling

Prevents any single domain from monopolizing synthesis cycles. If one domain is growing much faster than others, throttling kicks in to give other domains a chance. Prevents runaway feedback loops where one topic dominates the entire graph.

Contributor Exemptions

KB-ingested nodes, manual seeds, and human contributors are exempt from concentration checks. The throttle only protects against runaway autonomous cycles — not against intentional human or bulk-import activity.

Junk Filter Decay

Junk embeddings decay over 30 days. Without decay, junking a node permanently blocks all future content about the same topic — even good content. The decay ensures the junk filter protects against recent bad patterns without permanently poisoning topics.

Birth / Cull Architecture

Synthesis uses a two-phase quality model: permissive birth + periodic culling. This separates creative mixing (which needs freedom to explore) from quality enforcement (which needs to be strict).

Birth (Mechanical + Minitruth)

Mechanical gates run first: resonance threshold, similarity ceiling, structural validation, truncation, word count, dedup, junk filter, specificity. Then minitruth evaluates the synthesis (accept / rework with feedback / reject) before the node enters the graph.

Cull (Single LLM Evaluation)

The Population Control cycle evaluates nodes after a grace period using a single comprehensive LLM call that scores coherence, grounding, novelty, specificity, and incremental value. Nodes that fail are demoted or archived.

See Verification & Quality for full gate details.

Population Control Cycle

An autonomous cycle that runs in the background, evaluating recently-synthesized nodes using a single comprehensive LLM call that scores coherence, grounding, novelty, specificity, and value.

Grace Period

Nodes get a configurable grace period (default 2 hours) after creation before facing evaluation. This lets new nodes participate in synthesis before being judged.

Single LLM Evaluation

One comprehensive LLM call per node scores 6 dimensions (coherence, grounding, novelty, specificity, forced analogy, incremental value). The weighted composite determines the outcome.

Outcomes

Boost (score ≥ threshold): weight increased. Demote (between thresholds): weight halved. Archive (below archive threshold): removed from active graph.

Key Parameters

Security & Remote Access

Security & Remote Access

Podbit has a layered security system designed for both localhost (zero-friction development) and remote access (network-exposed server with full authentication). When running locally, security is transparent. When exposed to a network, JWT authentication, TLS, CORS lockdown, rate limiting, and security headers activate automatically. See the Installation & Setup page for how to get Podbit running.

Localhost Mode (Default)

  • Server binds to 127.0.0.1
  • GUI authenticates via automatic key handshake
  • No password required, no login screen
  • Proxy accepts all requests without auth
  • CORS allows all origins

This is the default experience. Nothing to configure.

Remote Mode

  • Server binds to 0.0.0.0 (set HOST=0.0.0.0)
  • JWT authentication required for all API requests
  • Admin password must be set on first access
  • Proxy requires security key or JWT bearer token
  • CORS locked to same-origin (or configured origins)
  • Security headers enforced (HSTS when TLS active)

Activates when HOST is not 127.0.0.1 or localhost.

JWT Authentication

Remote mode uses JWT access tokens (HMAC-SHA256) signed with the security key. Zero external auth dependencies - built entirely on Node.js crypto.

Access Token
  • 15-minute TTL
  • Sent as Authorization: Bearer <token>
  • Stored in sessionStorage (browser)
  • HMAC-SHA256 signature, base64url encoding
Refresh Token
  • 7-day TTL
  • Delivered as httpOnly cookie (XSS-resistant)
  • SHA-256 hash stored in DB (not the raw token)
  • Family-based rotation with theft detection

Refresh Token Theft Detection

Each refresh token belongs to a family (per RFC 6749 §10.4). When a token is refreshed, the old one is revoked and a new one issued in the same family. If a revoked token is ever reused (indicating theft), the entire family is revoked - forcing re-login on all sessions from that family.

Admin Password Setup

On first access in remote mode, the GUI prompts you to set an admin password. The password is hashed with scrypt (N=16384, r=8, p=1) and stored in the system database.

Implementation Notes

  • Zero auth dependencies - JWT signing, token hashing, and password hashing all use Node.js built-in crypto module (except cookie-parser for httpOnly cookie handling)
  • Refresh tokens in DB - stored as SHA-256 hashes in system.db, so a database compromise doesn't leak raw tokens
  • Periodic cleanup - expired refresh tokens are automatically purged every 6 hours
  • Password changes - changing the admin password or regenerating the security key revokes all active refresh tokens, forcing re-login everywhere
  • GUI auto-refresh - the GUI client automatically refreshes expired access tokens using the httpOnly refresh cookie, with concurrent request queuing to avoid token races

Weight Mechanics

Weight is adjusted additively or subtractively by multiple signals — ratings, filters, LLM judges, and cycle-based decay. Nodes start at 1.0. There is no hard upper bound; a floor of 0.1 prevents nodes from dropping to zero.

+0.1 synthesis boostParent nodes gain +0.1 each time one of their children is created via synthesis
+0.15 / +0.05 ancestryWhen a descendant becomes breakthrough: parent +0.15, grandparent +0.05
+0.2 / −0.1 / −0.3 feedbackHuman or agent ratings: useful +0.2, not useful −0.1, harmful −0.3
+0.05 context usageNodes the LLM actually references in a chat response get a small weight boost per turn
± lab confidenceLab verification adjusts weight by +0.15 × confidence (verified) or −0.15 × confidence (disproved)
± autoratingAutonomous LLM judge scores quality and applies a delta — rewards specific, grounded content; penalises vague output
×0.999 periodic decayAll weights multiplied by the decay factor every N cycles — prevents old content from dominating indefinitely
× extra synthesis decaySynthesis and voiced nodes never referenced by the context engine accumulate an additional decay multiplier after a grace period

Salience Mechanics

Salience is an attention score ranging from 0.01 (floor) to 1.0 (ceiling) that controls how often a node is sampled for synthesis pairing.

Boost on match+0.1 when a node is selected and paired during synthesis
Decay per cycleMultiplied by 0.99 per N cycles to prevent stale dominance
High salienceFrequently sampled for synthesis pairing
Low salienceRarely sampled, giving room for fresh nodes to participate

Trajectory Classification

Every synthesis node is classified into one of two trajectories based on automatic content analysis.

Knowledge

Specific, factual content containing numbers, technical terms, and concrete claims. Parents of knowledge-trajectory breakthroughs receive weight boosts.

Abstraction

General, philosophical content. Receives lower initial weight to prevent vague drift and keep the graph grounded in concrete knowledge.

Embeddings

Every node is represented as a vector embedding — a high-dimensional numeric representation of its content.

  • •Configurable dimensions (768 default)
  • •Computed by the embedding subsystem (local models recommended for speed and cost)
  • •Used for: similarity search, synthesis pairing, dedup detection, relevance scoring
  • •In-memory computation, efficient up to ~10K nodes

Structured description of what to test, extracted from a claim by Podbit. Sent to labs — never contains the raw claim narrative.
Falsifiability Review
Adversarial LLM check on extracted specs. A second model judges whether setup parameters are cherry-picked to guarantee the claimed result. Catches tautological tests that pass structural checks.
Freeze
Node state during active lab verification — frozen nodes are excluded from synthesis, decay, and lifecycle sweeps.
Fitness grading
Quality gate assessing overall synthesis output quality.
Gold standard

Hallucination detection
Quality gate flagging invented information in synthesis output.
Heuristic pipeline

Junk filter
